## Переходим к Этапу 2.5: Проектирование формата сообщений Kafka (JSON Schema)

Наш Python-парсер будет собирать вакансии и отправлять их в топик Kafka (назовем его `vacancies-raw`). Go-сервис будет читать эти сообщения. Им нужен единый, понятный обоим языкам формат данных.

Поскольку данные структурированные, мы будем использовать формат **JSON**.

Давайте спроектируем схему этого сообщения. Нам нужно передавать:
1. `title` (строка) — Название вакансии.
2. `company` (строка) — Название компании (всегда "VK" на данном этапе).
3. `city` (строка) — Город.
4. `url` (строка) — Ссылка на вакансию (наш уникальный ключ).
5. `direction` (строка) — Направление с сайта (пока "backend-разработка").
6. `scraped_at` (строка, формат ISO 8601) — Время парсинга, чтобы отслеживать актуальность.

## Техническая спецификация

### Схема базы данных
Таблица `vacancies` инициализируется через SQL-миграции в папке `/migrations`.

### Формат сообщений в Kafka
Парсер на Python отправляет найденные вакансии в топик `vacancies-raw` в виде JSON-сообщений следующего формата:

#### Пример:
```json
{
  "title": "Junior Go Developer",
  "company": "VK",
  "city": "Москва",
  "url": "https://education.vk.company/vacancy/123",
  "direction": "backend-разработка",
  "scraped_at": "2024-03-07T15:30:00Z"
}
```

#### Описание полей:
* `title` (string, required): Название вакансии.
* `company` (string, required): Компания-работодатель.
* `city` (string, required): Город работы.
* `url` (string, required, unique): Уникальная ссылка на вакансию.
* `direction` (string, required): Направление вакансии.
* `scraped_at` (string, required): Таймштамп парсинга в формате ISO 8601 UTC.
